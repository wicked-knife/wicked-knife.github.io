# 代码拆分

webpack的代码拆分功能可以将打包出的 bundle 拆分成体积更小的 bundle。

这是代码打包优化的主要手段，运用合理可以极大减少应用加载时间，提升应用性能。

常用的代码分离方法有三种：

  • 入口起点：使用 entry 配置手动地分离代码。

  • 防止重复：使用 SplitChunksPlugin 去重和分离 chunk。

  • 动态导入：通过模块中的内联函数调用来分离代码。

# 入口起点

这种方式主要通过 entry 手动显示地告诉webpack那些代码要分开打包。

在webpack.common.js中作如下配置:

module.exports = {
    ...
    entry: {
      index: './index.js',
      moduleA: './moduleA.js'
    }
    ...
}

# 防止重复

入口起点的方式直观，简单。但是不够灵活，无法实现动态地拆分代码。如果index.js 和 moduleA.js 同时引用一个公共模块，如'lodash'时， webpack会将lodash同时打包进去，这样造成了代码冗余。

使用SplitChunksPlugin就可以将模块的公用依赖拆分到一个单独的模块中。

在webpack.common.js中作如下配置:

module.exports = {
    ...
    optimization: {
      splitChunks: {
        chunks: 'all'
      }
    }
    ...
}

这样就可以满足以上需求。

# 动态加载

使用动态加载的方式分割代码主要使用import语法配合magic comment来是实现。

因为我们的代码使用了babel编译，所以为了让babel识别动态import的语法，需要安装@babel/plugin-syntax-dynamic-import

npm i -D @babel/plugin-syntax-dynamic-import

.babelrc配置:

{
  ...
  "plugins": ["@babel/plugin-syntax-dynamic-import"]
  ...
}

在webpack.common.js中作如下配置:

module.exports = {
    ...
    output: {
      chunkFilename: '[name].bundle.js'
    }
    ...
}

chunkFilename: 它决定 non-entry chunk(非入口 chunk) 的名称。即通过动态加载拆分的代码块的名称。

这个时候假如我们想动态引入'lodash':

// index.js
import(/*webpackChunkName: "lodash"*/ 'lodash').then(({default:　_}) => {
  // you can use lodash as _
})

这样即可动态引入lodash。同时因为我们配置了chunkFileName,并且在引入时提供了'webpackChunkName',在打包时lodash就会自动拆分出来，并命名为 'lodash.bundle.js' 。

很多时候我们也有 prefetch 和 preload 模块的需求。

一个常见的场景就是在用户网络空闲时去加载代码。

  • prefetch(预取)：将来某些导航下可能需要的资源
  • preload(预加载)：当前导航下可能需要资源

使用方法：

import(/* webpackPrefetch: true */ './LoginModal.js');

这会生成 并追加到页面头部，指示着浏览器在闲置时间预取 login-modal-chunk.js 文件。

与 prefetch 指令相比，preload 指令有许多不同之处：

  • preload chunk 会在父 chunk 加载时，以并行方式开始加载。prefetch chunk 会在父 chunk 加载结束后开始加载。
  • preload chunk 具有中等优先级，并立即下载。prefetch chunk 在浏览器闲置时下载。
  • preload chunk 会在父 chunk 中立即请求，用于当下时刻。prefetch chunk 会用于未来的某个时刻。
  • 浏览器支持程度不同。
